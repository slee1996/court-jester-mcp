# Court Jester Sprint Plan

Date: 2026-03-18

## Sprint Goal

Make `repair-loop` the primary Court Jester workflow and improve its ability to break AI-generated code quickly with actionable failures.

## Current State

Latest expanded benchmark snapshot from `/tmp/court-jester-bench-full-v10`:

- `claude-default`
  - `baseline`: `9 / 9`
  - `repair-loop`: `9 / 9`
  - `required-final`: `9 / 9`
- `codex-default`
  - `baseline`: `8 / 9`
  - `repair-loop`: `9 / 9`
  - `required-final`: `8 / 9`

Interpretation:

- `repair-loop` is currently the most promising product path.
- `required-final` is still useful as a control, but it is not the best default behavior.
- the remaining work should optimize hostile failure discovery and fast repair, not static-analysis polish.

## Prioritized Tasks

### P0. Make benchmark failures legible

1. Add structured failure categories to benchmark result rows.
Estimate: `0.5d`
Success:
- every `result.json` includes `failure_category`
- categories distinguish provider/auth issues, verify false positives, verify-caught bugs, public regressions, hidden semantic misses, and timeout-driven infra failures

2. Extend summary output to count failure categories by `model x policy`.
Estimate: `0.5d`
Success:
- `bench/summarize_runs.py` prints category counts that make repair-loop regressions easy to spot

3. Add sprint checkpoint reporting to docs.
Estimate: `0.25d`
Success:
- benchmark notes and sprint notes stay in repo, not only in `/tmp`

### P0. Strengthen repair-loop signal quality

4. Improve `verify` failure payloads for repair prompts.
Estimate: `1d`
Success:
- repair feedback includes failing input, stage, and shortest available repro
- provider feedback is concise enough to reuse directly in a repair turn

5. Make `repair-loop` the primary policy path in the harness.
Estimate: `0.5d`
Success:
- runner code clearly models `apply -> verify -> repair -> reverify`
- artifacts make each attempt easy to compare

### P1. Expand hostile input synthesis

6. Push Python nested-dict and sparse-shape fuzzing further.
Estimate: `1d`
Success:
- more coverage for nested `None`, partial maps, empty lists, and blank-present strings in config/profile-style inputs

7. Push TypeScript nullable object and sparse-array fuzzing further.
Estimate: `1d`
Success:
- more coverage for optional containers, `null` unions, blank strings, and boundary indexing bugs

8. Add direct regression tests for each newly discovered verifier miss.
Estimate: `0.5d`
Success:
- every benchmark-discovered blind spot gets a verifier regression test in `tests/verify_test.rs` or `tests/synthesize_test.rs`

### P1. Expand the adversarial benchmark

9. Grow the task suite from `15` to `30`.
Estimate: `2d`
Success:
- balanced Python/TypeScript coverage
- more hidden-check failures that public checks miss

10. Add a dedicated adversarial task bucket.
Estimate: `1d`
Success:
- tasks specifically target nested nullable data, malformed plausible objects, whitespace-only values, and array-boundary bugs

11. Keep dual-model coverage mandatory.
Estimate: `0.25d`
Success:
- every matrix run keeps `claude-default` and `codex-default` under `baseline`, `required-final`, and `repair-loop`

### P2. Keep infra noise out of the loop

12. Continue hardening runtime stability in sandbox/verify.
Estimate: `0.5d`
Success:
- new failures are dominated by semantic problems, not startup/tooling timeouts

## Proposed Sprint Order

1. P0.1 structured failure categories
2. P0.2 summary category counts
3. P0.4 better repair payloads
4. P0.5 repair-loop-first runner cleanup
5. P1.6 and P1.7 synthesis expansion
6. P1.8 regression locking
7. P1.9 and P1.10 benchmark expansion
8. P2.12 runtime stabilization follow-ups if new infra failures appear

## Exit Criteria

- `repair-loop` beats baseline across the expanded suite
- false positives remain rare and explicit
- most remaining failures are real semantic misses or provider behavior, not harness noise
- Court Jester is demonstrably useful as an in-loop breaker, not just a checker
